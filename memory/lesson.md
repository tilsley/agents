
## lesson-802mzt — 2026-02-24

Problem: Automated security patch downgraded `@aws-amplify/ui-react` from 4.1.0 to 2.0.0 (a major version regression) while labeling it a vulnerability fix, causing a build failure due to a missing `./aws-exports` module — a breaking API/config change tied to Amplify major version compatibility. The `aws-exports.js` file is generated by the Amplify CLI and its expected location/format changes across major versions (v4→v5 requires renaming to `amplifyconfiguration.json`).
Solution: Automated patch agents must validate that version changes are upgrades, not downgrades (enforce semver monotonicity). For known ecosystem migrations like AWS Amplify v4→v5, maintain a checklist of required file/config changes (e.g., rename `aws-exports.js` to `amplifyconfiguration.json`, update import paths) that must be applied alongside the version bump. Patch agents should run `npm run build` in a sandbox and only open a PR if the build succeeds. After any major Amplify version change, re-run `amplify pull` or ensure config files are regenerated. Pin bot-generated PRs to require a passing build before merge, and require human review for any major version change regardless of the vulnerability justification.
Context: Applies to any automated dependency patching workflow (Snyk, Dependabot, Renovate) where the agent may resolve a vulnerability by selecting a lower semver that satisfies the CVE fix range but regresses functionality. Especially risky for tightly coupled SDK ecosystems like AWS Amplify, Firebase, or similar where multiple interdependent packages share configuration files. The `aws-exports` → `amplifyconfiguration.json` migration is documented in the Amplify v5 migration guide.
Outcome: Build failed to compile; PR received a 0/100 review score and was blocked for manual review. The downgrade introduced a runtime misconfiguration that would break the app in production if merged, and all 25 vulnerability fixes were left unresolved.

## lesson-kd03jr — 2026-02-24

Problem: A single automated PR bundled 25 vulnerabilities across 3 packages (`react-scripts`, `aws-amplify`, `@aws-amplify/ui-react`) with very different severity levels, upgrade complexity, and dependency trees, making triage, rollback, and root-cause analysis significantly harder. All fixes were blocked by a single build failure unrelated to most of the CVEs.
Solution: Scope automated security PRs to one package (or one dependency subtree) per PR, grouping at most by risk tier (e.g., critical/high CVEs with breaking API changes separated from routine patch-level bumps). This isolates build failures, makes review tractable, allows partial merges when some fixes are safe and others are not, and clearly attributes failures to specific changes. Establish a routine dependency update cadence (e.g., monthly minor/patch bumps) to avoid large, risky batch upgrades accumulating in the first place.
Context: Applies to automated security patching pipelines managing multiple vulnerable dependencies in a single repo, especially when packages have interdependencies (e.g., `aws-amplify` and `@aws-amplify/ui-react`) or when dependency versions have drifted many major versions behind (e.g., `react-scripts` at 2.6.3, `aws-amplify` at 0.21.4).
Outcome: All 25 vulnerability fixes were blocked by a single bad version resolution, achieving 0 security improvement despite significant automation effort. Catching drift early with routine cadence keeps individual PRs small and safer to validate automatically.

## lesson-cwnbaw — 2026-02-24

Problem: The `package-lock.json` lockfile version was changed from v1 to v3 as a side effect of the dependency update, indicating a Node.js/npm toolchain version mismatch between the patch agent environment and the project's established toolchain. Additionally, `react-scripts` was listed in `package.json` as `^5.0.0` but the installed version was 2.6.3, indicating the stale lockfile was not honoring the declared range.
Solution: Automated patch agents should regenerate lockfiles using the same npm/Node.js version as the project's CI environment. The lockfile version should be pinned in CI configuration (e.g., via `.nvmrc` or `engines` field in `package.json`), and any lockfile version change should trigger a warning or block automatic merge. Regularly regenerate `package-lock.json` to ensure installed versions match declared ranges. Add a CI check that verifies the lockfile is consistent with `package.json` (e.g., `npm ci --dry-run` or `npm ls` integrity check).
Context: Node.js projects with long-lived lockfiles (v1) where automated tooling runs on newer npm versions that produce v3 lockfiles, potentially causing peer resolution differences in downstream installs. Also applies when `package-lock.json` has been committed at an old lockfile version and not refreshed, causing installed versions to drift from `package.json` ranges.
Outcome: Stale lockfile masked outdated transitive dependencies, allowing vulnerabilities to persist despite declared version ranges being satisfied by safe versions. Lockfile format regression went undetected as a separate issue because it was overshadowed by the build failure; it could silently cause resolution differences in other environments.
